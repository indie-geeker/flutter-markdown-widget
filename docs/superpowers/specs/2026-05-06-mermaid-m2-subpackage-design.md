## 1. Goal

Ship `flutter_markdown_widget_mermaid`, an independent pub package that provides the default `MermaidRenderer` implementation backed by `flutter_js` + bundled `mermaid.js`. After M2, end users can opt into Mermaid rendering with a one-line `pub add` and zero further configuration.

### Success criteria

- A user can `flutter pub add flutter_markdown_widget flutter_markdown_widget_mermaid` and see Mermaid diagrams render on iOS, Android, and macOS without any additional setup.
- The subpackage is fully decoupled from main-package internals: only depends on `MermaidRenderer` interface + public Mermaid types.
- Real-platform smoke tests run in CI on iOS / Android / macOS.
- Asset integrity verified via SHA-256 checksum on every CI run.
- Zero breaking change to main package (M2 is purely additive, separately published).
- Subpackage main-package version pin is enforced (`^0.2.0`) so future main-package upgrades fail closed rather than silently drift.

### Non-goals

- Web-platform native Mermaid rendering (deferred — see §6.5 item 1).
- Pure-Dart Mermaid implementation.
- Multi-renderer comparisons / A/B switching infrastructure.
- mermaid.js 11.x support (M2 targets 10.x; 11 is a single-PR upgrade later — §6.5 item 2).
- Embedded `MermaidRenderer` running in a Dart isolate (deferred — §6.5 item 3).

### 2.3 Public API surface

```dart
// lib/flutter_markdown_widget_mermaid.dart
export 'src/flutter_js_mermaid_renderer.dart' show FlutterJsMermaidRenderer;
```

Single class exported. All other types under `src/` are subpackage-private.

```dart
class FlutterJsMermaidRenderer implements MermaidRenderer {
  /// Default path: bundled mermaid.js, process-global singleton.
  /// Returns an [_UnsupportedRenderer] stub on Web (kIsWeb).
  factory FlutterJsMermaidRenderer.shared();

  /// Advanced path: user-supplied mermaid.js. Each call returns an
  /// independent instance (not shared).
  factory FlutterJsMermaidRenderer.withCustomMermaidJs({
    required String mermaidJs,
    required String mermaidVersion,
  });

  // MermaidRenderer interface (M1)
  @override Future<MermaidArtifact> render(String source, {required MermaidTheme theme});
  @override bool get isReady;
  @override String get version;
  @override Future<void> dispose();
}
```

The constructor `_internal` and any test-only entry points (`@visibleForTesting forTesting`) are not part of the public API.

### 2.4 External dependencies

The subpackage depends on **exactly two non-Flutter packages** at runtime:

- `flutter_markdown_widget: ^0.2.0` — for `MermaidRenderer`, `MermaidArtifact`, `MermaidTheme`, `MermaidError` subclasses.
- `flutter_js: ^0.x.y` — exact version locked at plan task 0 (open question, §6.7).
- `meta: ^1.16.0` — for `@visibleForTesting`.

The subpackage **does not** import any `lib/src/...` path from the main package. Boundary enforcement: a CI step greps `lib/` for forbidden imports.

## 3. JS engine integration & lifecycle

### 3.1 Render call stack

```
M1: MermaidView triggers render
       ↓
FlutterJsMermaidRenderer.render(source, theme: T)
       ↓
1. _ensureInitialized()                     ← lazy
       └─ first call only:
            mermaidJs = await _loader.load()
            _runtime = getJavascriptRuntime()    // flutter_js
            _runtime.evaluate(mermaidJs)
            _runtime.evaluate(mermaidInitJs)     // securityLevel: strict
       ↓
2. injected = MermaidThemeDirective.inject(source, theme)
       ↓
3. result = await _runtime.evaluateAsync(
            'mermaid.render(id, source).then(r => ({svg: r.svg}))
                                       .catch(e => ({error: e.message}))')
       ↓
4. parse JSON → either { svg } or { error }
       ↓
5. svg = MermaidSvgPostprocessor.injectBackground(rawSvg, theme)
       ↓
6. intrinsicSize = MermaidArtifact.parseViewBox(svg)
       ↓
7. return MermaidArtifact(svg: svg, intrinsicSize: intrinsicSize)
```

### 3.2 Lazy init protocol (concurrent-safe)

```dart
class FlutterJsMermaidRenderer implements MermaidRenderer {
  JavascriptRuntime? _runtime;
  Completer<void>? _initCompleter;
  bool _initFailed = false;
  Object? _initFailureCause;

  Future<void> _ensureInitialized() async {
    if (_initFailed) {
      throw MermaidInitializationError(
        source: '',
        cause: _initFailureCause!,
        stackTrace: StackTrace.current,
      );
    }
    if (_runtime != null) return;

    if (_initCompleter != null) {
      // Concurrent caller — share the same in-flight init.
      await _initCompleter!.future;
      return;
    }

    _initCompleter = Completer<void>();
    try {
      final mermaidJs = await _loader.load();
      final rt = getJavascriptRuntime();
      rt.evaluate(mermaidJs);
      rt.evaluate(_initJs);  // mermaid.initialize({...})
      _runtime = rt;
      _initCompleter!.complete();
    } catch (e, st) {
      _initFailed = true;
      _initFailureCause = e;
      _initCompleter!.completeError(e, st);
      throw MermaidInitializationError(source: '', cause: e, stackTrace: st);
    }
  }

  static const String _initJs =
      'mermaid.initialize({ startOnLoad: false, securityLevel: "strict" });';
}
```

**Key invariant**: `_initCompleter` is **never reset**. Once init succeeds, `_runtime != null` short-circuits the protocol. Once init fails, `_initFailed` short-circuits with synchronous `MermaidInitializationError`.

### 3.3 Render method

```dart
@override
Future<MermaidArtifact> render(String source, {required MermaidTheme theme}) async {
  await _ensureInitialized();   // may throw MermaidInitializationError

  final injected = MermaidThemeDirective.inject(source, theme);
  final id = 'mermaid_${_idCounter++}';

  final js = '''
    mermaid.render(${jsonEncode(id)}, ${jsonEncode(injected)})
      .then(r => JSON.stringify({ svg: r.svg }))
      .catch(e => JSON.stringify({ error: (e && e.message) || String(e) }))
  ''';

  final JsEvalResult result;
  try {
    result = await _runtime!.evaluateAsync(js);
  } catch (e, st) {
    throw MermaidRuntimeError(source: source, cause: e, stackTrace: st);
  }

  final parsed = jsonDecode(result.stringResult) as Map<String, dynamic>;
  if (parsed['error'] != null) {
    throw MermaidSyntaxError(
      source: source,
      message: parsed['error'] as String,
      stackTrace: StackTrace.current,
    );
  }

  final rawSvg = parsed['svg'] as String;
  final processedSvg = MermaidSvgPostprocessor.injectBackground(rawSvg, theme);
  final intrinsicSize = MermaidArtifact.parseViewBox(processedSvg);

  return MermaidArtifact(svg: processedSvg, intrinsicSize: intrinsicSize);
}
```

### 3.4 Error mapping (matches M1 spec §5.1)

| Subpackage condition | Maps to |
|----------------------|---------|
| `evaluate(mermaidJs)` throws during init | `MermaidInitializationError` (permanent state) |
| `evaluateAsync(...)` future throws | `MermaidRuntimeError` |
| `mermaid.render` Promise rejects with parse-error message | `MermaidSyntaxError(message: ...)` |
| Returned SVG missing `<svg>` tag or invalid XML | M1 widget layer triggers `MermaidInvalidOutputError` (subpackage does not throw this) |
| Render exceeds `MermaidOptions.renderTimeout` | M1 widget wraps with `Future.timeout` → `MermaidTimeoutError`. Subpackage is unaware. |

### 3.5 Dispose semantics

```dart
@override
Future<void> dispose() async {
  _runtime?.dispose();
  _runtime = null;
  _initFailed = true;
  _initFailureCause = StateError('renderer disposed');
}
```

After `dispose()`, `isReady` returns `false`, and `render()` throws `MermaidInitializationError`. Matches M1 spec §3.5 contract.

### 3.6 isReady / version

```dart
@override
bool get isReady => _runtime != null && !_initFailed;

@override
String get version =>
    'flutter-js-${packageVersion}+${_loader.sourceVersion}';
//  Example: "flutter-js-0.1.0+mermaid-10.6.0"
```

`packageVersion` is generated at build time from `pubspec.yaml` into `_package_version.dart`. CI runs the generator before tests; `.gitignore` excludes the generated file.

## 4. mermaid.js bundling & injection

### 4.1 `MermaidJsLoader` abstraction

```dart
abstract class MermaidJsLoader {
  Future<String> load();
  String get sourceVersion;
}
```

Two implementations:

#### `_BundledMermaidJsLoader`

```dart
class _BundledMermaidJsLoader implements MermaidJsLoader {
  static const String bundledVersion = '10.6.0';
  static String? _cached;       // process-level cache

  @override
  Future<String> load() async {
    if (_cached != null) return _cached!;
    _cached = await rootBundle.loadString(
      'packages/flutter_markdown_widget_mermaid/assets/mermaid.min.js',
    );
    return _cached!;
  }

  @override
  String get sourceVersion => 'mermaid-$bundledVersion';
}
```

**Process-level cache**: even with multiple `FlutterJsMermaidRenderer` instances, mermaid.js is loaded from the asset bundle only once.

#### `_InjectedMermaidJsLoader`

```dart
class _InjectedMermaidJsLoader implements MermaidJsLoader {
  _InjectedMermaidJsLoader(this._source, this._userVersion);
  final String _source;
  final String _userVersion;

  @override
  Future<String> load() async => _source;

  @override
  String get sourceVersion => 'mermaid-$_userVersion';
}
```

### 4.2 Internal `_JsRuntime` abstraction

Tests must run in the dart vm without `flutter_js`'s native bindings. The subpackage therefore defines its own minimal interface and wraps flutter_js's `JavascriptRuntime` with an adapter. The interface is **subpackage-private** (single underscore) — not exposed to consumers.

```dart
// lib/src/_js_runtime.dart  (private)
abstract class _JsRuntime {
  /// Synchronously evaluates [code] for side effects (init).
  void evaluate(String code);

  /// Asynchronously evaluates [code] and returns the resulting string.
  /// Implementations resolve when the returned JS Promise settles.
  Future<String> evaluateAsync(String code);

  void dispose();
}

class _FlutterJsRuntime implements _JsRuntime {
  _FlutterJsRuntime() : _rt = getJavascriptRuntime();
  final JavascriptRuntime _rt;

  @override
  void evaluate(String code) => _rt.evaluate(code);

  @override
  Future<String> evaluateAsync(String code) async {
    final result = await _rt.evaluateAsync(code);
    return result.stringResult;
  }

  @override
  void dispose() => _rt.dispose();
}
```

Tests implement `_JsRuntime` directly (`_FakeJsRuntime`) without depending on any flutter_js types, keeping unit tests `dart vm`–only and fast.

### 4.3 Two factory paths

```dart
class FlutterJsMermaidRenderer implements MermaidRenderer {
  FlutterJsMermaidRenderer._internal({
    required MermaidJsLoader loader,
    required _JsRuntime Function() runtimeFactory,
  })  : _loader = loader,
        _runtimeFactory = runtimeFactory;

  static FlutterJsMermaidRenderer? _shared;

  factory FlutterJsMermaidRenderer.shared() {
    if (kIsWeb) return _UnsupportedMermaidRenderer();
    return _shared ??= FlutterJsMermaidRenderer._internal(
      loader: _BundledMermaidJsLoader(),
      runtimeFactory: () => _FlutterJsRuntime(),
    );
  }

  factory FlutterJsMermaidRenderer.withCustomMermaidJs({
    required String mermaidJs,
    required String mermaidVersion,
  }) {
    if (kIsWeb) return _UnsupportedMermaidRenderer();
    return FlutterJsMermaidRenderer._internal(
      loader: _InjectedMermaidJsLoader(mermaidJs, mermaidVersion),
      runtimeFactory: () => _FlutterJsRuntime(),
    );
  }

  @visibleForTesting
  factory FlutterJsMermaidRenderer.forTesting({
    required MermaidJsLoader loader,
    required _JsRuntime Function() runtimeFactory,
  }) =>
      FlutterJsMermaidRenderer._internal(
        loader: loader,
        runtimeFactory: runtimeFactory,
      );
}
```

Note: `_JsRuntime` being subpackage-private but exposed via `@visibleForTesting forTesting` means **only subpackage tests** can implement fakes — third-party consumers cannot reach it. This is intentional: tests outside the subpackage should mock the public `MermaidRenderer` interface (M1's `FakeMermaidRenderer`), not the JS layer.

**Why custom-mermaid.js path is not shared**: a user calling `withCustomMermaidJs` twice (e.g., for A/B testing of mermaid versions) expects two independent runtimes. Sharing would silently merge them — surprising and unsafe.

## 5. Theme injection & SVG post-processing

### 5.1 `MermaidThemeDirective`

```dart
abstract final class MermaidThemeDirective {
  static String inject(String source, MermaidTheme theme) {
    assert(theme != MermaidTheme.auto,
        'auto must be resolved before reaching the renderer');
    final mermaidName = _mapTheme(theme);

    if (_existingInitPattern.hasMatch(source)) {
      return source;        // user-supplied directive wins
    }

    return '%%{init: {"theme":"$mermaidName"}}%%\n$source';
  }

  static String _mapTheme(MermaidTheme theme) => switch (theme) {
    MermaidTheme.light => 'default',
    MermaidTheme.dark => 'dark',
    MermaidTheme.neutral => 'neutral',
    MermaidTheme.forest => 'forest',
    MermaidTheme.auto => 'default',   // defensive fallback
  };

  static final RegExp _existingInitPattern =
      RegExp(r'^\s*%%\{\s*init\s*:', multiLine: false);
}
```

**Per-call directive**, not global `mermaid.initialize()` — multiple Mermaid blocks with different themes render correctly without state pollution.

**User init directive wins**: if the source already contains `%%{init: ...}%%`, the inject is skipped. Documented behavior, prevents undefined precedence.

### 5.2 `MermaidSvgPostprocessor`

mermaid.js outputs SVG with a transparent root. On a Material light card, dark-theme SVG (light text) is invisible. Solution: inject a 100%-sized `<rect>` immediately after the `<svg>` open tag.

```dart
abstract final class MermaidSvgPostprocessor {
  static String injectBackground(String svg, MermaidTheme theme) {
    if (_alreadyHasBackground(svg)) return svg;

    final color = _backgroundColor(theme);
    final match = _svgOpenTag.firstMatch(svg);
    if (match == null) return svg;   // not valid SVG → upper layer detects
    final insertAt = match.end;
    return '${svg.substring(0, insertAt)}'
           '<rect width="100%" height="100%" fill="$color"/>'
           '${svg.substring(insertAt)}';
  }

  static String _backgroundColor(MermaidTheme theme) => switch (theme) {
    MermaidTheme.light => '#ffffff',
    MermaidTheme.dark => '#1f2020',
    MermaidTheme.neutral => '#f8f9fa',
    MermaidTheme.forest => '#f4f4f4',
    MermaidTheme.auto => '#ffffff',
  };

  static bool _alreadyHasBackground(String svg) =>
      _existingBackgroundPattern.hasMatch(svg);

  static final RegExp _svgOpenTag =
      RegExp(r'<svg\b[^>]*>', caseSensitive: false);

  static final RegExp _existingBackgroundPattern =
      RegExp(r'<rect[^>]*width="100%"[^>]*height="100%"', caseSensitive: false);
}
```

**Why post-processing lives in M2 (not M1 widget)**: M1's `MermaidArtifactView` is intentionally theme-agnostic. The mermaid-color-conventions knowledge belongs to whoever generates the SVG. If a future renderer supplies its own background, M2's post-processor short-circuits via `_alreadyHasBackground`.

### 5.3 No user override in M2

`MermaidOptions.svgBackgroundOverride` is **not** added in M2. If users need transparent SVG (e.g., for PDF export), it goes through the M1 widget layer in a future release.

### 5.4 SVG post-process failure mode

If `injectBackground` cannot find `<svg>` (returned content is not valid SVG), the function returns the original string unmodified. `MermaidArtifact.parseViewBox` will then return `null` for `intrinsicSize`, and downstream M1 `flutter_svg` will fail and trigger `MermaidInvalidOutputError` (M1 spec §5.1 F5). The subpackage does not actively throw on malformed SVG.

### 6.3 Real-platform smoke

`integration_test/mermaid_smoke_test.dart`:

```dart
import 'package:flutter_test/flutter_test.dart';
```

### 6.6 Boundary-enforcement test

A `test/boundary_test.dart` script greps the subpackage's `lib/` for forbidden imports:

```dart
test('subpackage does not import main package internals', () async {
  final files = Directory('lib').listSync(recursive: true)
      .whereType<File>()
      .where((f) => f.path.endsWith('.dart'));
  for (final file in files) {
    final content = await file.readAsString();
    expect(
      content.contains("package:flutter_markdown_widget/src/"),
      isFalse,
      reason: '${file.path} imports main package internals — use public API only',
    );
  }
});
```

This catches accidental `lib/src/...` leaks early.

### 7.2 Cross-repo development workflow

During M2.1 / M2.2, the subpackage uses a path dependency:

```yaml
dependencies:
  flutter_markdown_widget:
    path: ../flutter-markdown-widget
```

**Convention**: both repos cloned side-by-side under the same parent directory. CI replicates this layout (§6.4 checkout step).

**Tag pinning prerequisite (must happen before M2.1 begins)**:

```bash
# In flutter-markdown-widget (main package) repo, on main branch at commit c6966d1
git tag -a v0.2.0-pre -m "M1 final state; pinned by flutter-markdown-widget-mermaid CI"
git push origin v0.2.0-pre
```

This tag is the M2 development reference: subpackage CI checks out this exact tag, isolating subpackage work from main-package drift. The owner (you) creates this tag manually as the very first step of M2 — before any subpackage code exists. Plan task 0 explicitly verifies the tag is pushed.

**M2.3 cutover**: a single PR changes:

1. `pubspec.yaml`: path → `^0.2.0`
2. CI: drop the second `actions/checkout` step (no longer need main repo locally)
3. `flutter pub publish --dry-run` clean
4. Tag `v0.1.0`
5. `flutter pub publish`

### 7.3 Compatibility

- **Zero breaking change** to `flutter_markdown_widget`: M2 only consumes the public M1 interface.
- **`@experimental` removal**: after M2.3 + one release cycle (~2-4 weeks) without user-reported interface issues, main package publishes `0.3.0` removing the `@experimental` annotation from `MermaidRenderer`.
- **Caret constraint discipline**: subpackage pins `flutter_markdown_widget: ^0.2.0`, allowing `0.2.x` updates but **not** `0.3.0` or higher. Future main-package minor bumps explicitly require a subpackage release.

## 8. Deferred work

In priority order, with origin-spec reference where applicable:

1. **Web platform native support** (M1 spec §8 item 1) — `dart:js` interop or `<script>` injection so mermaid.js runs in the browser main thread.
2. **mermaid.js 11.x upgrade** — separate PR; major version diff means review of theme variables, init API, output SVG structure.
3. **Renderer in Dart isolate** (M1 spec §8 item 8) — benchmark first to verify UI thread blocking is observable.
4. **mermaid `themeVariables` / `themeCSS`** (M1 spec §8 item 4) — beyond the four built-in themes.
5. **Pre-rendered / SSR mode** (M1 spec §8 item 7) — build-time SVG embedded in markdown front-matter; runtime reads directly.
6. **Multiple renderer instance cache key isolation** — when a user creates several `withCustomMermaidJs` instances, M1's `MermaidCache` should not collide between them. Current cache key includes `rendererVersion`, but two instances with the same custom mermaid string have identical version strings.
7. **Streaming mermaid render in isolate** — non-blocking flow for very large mermaid sources.
8. **iOS/Android memory pressure hook** — listen for `MemoryWarning` to clear mermaid.js's internal cache.
9. **`MermaidOptions.svgBackgroundOverride: Color?`** — user-controlled SVG background (e.g., for PDF export with transparency).
10. **Subpackage's own example app published as a demo** — showing diagram types beyond smoke fixture.

## 9. Known risks

| Risk | Mitigation | Trigger response |
|------|------------|------------------|
| `flutter_js` becomes incompatible with future Flutter SDK | Renderer interface is pluggable; subpackage isolated from main | Switch backing to `flutter_qjs` or fork; only subpackage code changes |
| mermaid 11 SVG output structure breaks our post-processor | Subpackage minor-bumps + smoke matrix runs full suite | Pin to mermaid 10 in production; defer 11 upgrade until issues resolved |
| iOS JavaScriptCore version fragmentation | Document minimum iOS support (matches Flutter default, iOS 13+) | Investigate per user bug report |
| Asset size bloat (mermaid 11 minified ~3MB) | `flutter build apk --analyze-size` step in release CI | Trigger deferred item 5 (SSR) or split mermaid.js by diagram type |
| Supply-chain compromise of mermaid.js CDN | sha256 checksum + manual review on every upgrade PR | Roll back to last known-good version; investigate publisher chain |
| Subpackage CI accidentally depends on main-package internals | Boundary-enforcement test (§6.6) | CI fails fast; PR cannot merge |
| Path-dependency drift between dev and CI | CI checks out tag `v0.2.0-pre`; same as developer convention | Document conventions in subpackage README |

## 10. Open questions for the implementation plan

1. **Exact `flutter_js` version**: pub.dev current latest, target is the most recent stable version compatible with Flutter 3.24+. Plan task 0 verifies and locks.
2. **Bundled `mermaid.js` version**: `10.6.0` is the placeholder. Plan task 0 verifies it's the latest 10.x stable (or selects appropriate alternative).
3. **iOS minimum target**: `flutter_js` documentation should confirm. If matches Flutter's default (iOS 13+), no extra constraint in pubspec. Otherwise add to README compatibility section.
4. **`securityLevel: "strict"` vs `"loose"`**: M2 starts with mermaid.js default (strict, disallows HTML in node labels). Switch to `loose` only if user reports concrete missing functionality.
5. **`unsupported` version string**: `_UnsupportedMermaidRenderer.version` exact value is implementation-detail; should be stable so cache misses don't churn. Suggest `'unsupported'` literally; plan resolves.
6. **`packageVersion` generation strategy**: build_runner script vs hand-maintained constant vs codegen step in CI. Plan task 0 picks one (recommend: simple sed in CI, no build_runner overhead).
